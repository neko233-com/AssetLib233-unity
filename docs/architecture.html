架构设计

AssetGroup 是边界，Manifest 是事实，Provider 是平台差异出口。本文档详细描述 AssetLib233 的分层架构、核心子系统和数据流。

1. 系统总览

AssetLib233 是一个单线程异步资源热更库。所有运行时操作通过主线程 Tick() 驱动，不使用线程、锁或并发容器，天然兼容 WebGL、微信小游戏、抖音小游戏、TapTap 小游戏等受限运行时。

2. 分层架构

AssetLib233 分为五层，从上到下依赖方向单一，禁止反向引用。

图 2：五层架构 — 应用层 → 门面层 → 子系统层 → 平台抽象层 → 运行时引擎层

层职责关键类型
应用层业务逻辑、启动编排、UI Loading项目代码、上层资源管理器

4. 下载子系统

下载子系统由 AssetLib233DownloadScheduler 驱动，支持单组下载和多组统一进度。

图 4：下载子系统 — Operation → Scheduler (Pending/Running/Finished/Failed) → Transport

关键特性

  • 并发控制：默认 10 并发，同 fileName 去重避免重复下载
  • 失败重试：失败请求进入 Retry Queue，延迟 1s 后重新调度
  • Fallback URL：主 CDN 失败自动切换备用地址
  • 进度聚合：UnifiedLoadingProgress 将多组下载汇总为单一进度条
  • 磁盘预检：下载前检查剩余空间是否充足

7. 平台插件体系

平台插件通过 IAssetLib233PlatformPlugin 接口隔离平台差异。每个小游戏平台有独立 asmdef 和编译宏，未定义宏时不参与编译。

图 7：平台插件体系 — PluginRegistry 管理四个内置插件，每个插件通过统一接口注入平台能力

10. 端到端数据流

从 CDN 到渲染完成，一个资源的完整生命周期：

CDN
  │  .version → .manifest / .bytes → .bundle
  ▼
PreparePackageOperation（版本 + Manifest 注入）
  │
  ▼
PackageDownloadOperation（差异 Bundle 下载 + 缓存写入）
  │
  ▼
LoadAssetAsync<T>（Manifest 查地址 → Bundle 加载 → Asset 提取）
  │
  ▼
AssetHandle233<T>（引用计数 Retain）
  │
  ▼
业务使用（handle.AssetObject）
  │
  ▼
handle.Release()（引用计数 Release → GC 回收 → Bundle 卸载）

11. 设计原则

原则说明
单线程异步所有运行时操作通过主线程 Tick() 驱动，不使用 Thread / Task.Run / 锁 / 并发容器
零 Runtime 依赖框架本体不依赖第三方库；UniTask 为可选插件
平台可插拔通过 IAssetLib233PlatformPlugin 隔离平台差异，新平台只需实现接口
CDN 向下兼容Runtime 可直接读取 YooAsset 标准 CDN 产物，不要求额外适配
稳定优先不为零 GC 牺牲可维护性和排障能力；热路径使用 NonAlloc / ListPool
AssetGroup 是边界框架不预设业务阶段名称；login / default / story 等完全由项目决定
Manifest 是事实所有加载、下载、缓存判定以 Manifest 为唯一真相源
Provider 是出口Editor / Runtime / 小游戏差异统一从 Provider 接口输出
